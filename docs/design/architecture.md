# Architecture

High-level overview of how KeenySpace is composed at runtime.

## Process composition

KeenySpace runs as a single process:

- **Single ASGI app** — FastAPI + FastMCP in one process via `app.mount("/v1/mcp", mcp.http_app(path="/"))` with mandatory `combine_lifespans(app_lifespan, mcp_app.lifespan)`. Serves `/v1/api/*`, `/v1/mcp`, and `/v1/admin/*`.
- **Compile agent** — pydantic-ai `Agent` that runs server-side as a background coroutine, triggered on demand (via REST/MCP) or by APScheduler. Runs inside the main process as an in-process background task (not a separate worker process).
- **Scheduler** — APScheduler 3.x for periodic compile triggers and retention tasks. Single worker in v1.

Multi-worker uvicorn is explicitly deferred to v1.5 (requires `fcntl.flock` + APScheduler dedupe + per-worker compile-agent isolation).

## LLM integration

All server-side LLM operations go through a single stack:

- **pydantic-ai** — agent framework: tool calls, model abstraction, structured outputs, observability.
- **instructor** — structured output extraction from LLM responses into Pydantic models.

Provider-agnostic: Anthropic (default), OpenAI (opt-in), or any OpenAI-compatible endpoint,
all through the pydantic-ai interface. The active provider is set via environment variables at
deploy time.

## Persistent state

| Storage | What |
|---------|------|
| FS root (`<server-data-root>/`) | `workspaces/<uuid>/` — markdown vaults; `blueprints/` — admin-only templates. Markdown files are the source of truth for content. |
| Postgres | Workspace registry (UUID to slug mapping, blueprint ref), users, sessions, API keys, ACL refs, audit log. Server-side metadata only; no page content. |

Single process, single FS root, single Postgres instance. Multi-machine replication is out of scope for v1.

## Self-hosted deployment (docker-compose)

The reference deployment ships as `deploy/docker-compose.yml` with:

- `keenyspace` — the server (Python 3.14, FastAPI + FastMCP)
- `postgres` — Postgres 17 (registry and sessions)
- `authentik` + `authentik-worker` — Authentik reference OIDC provider
- `caddy` — reverse proxy with auto-HTTPS and SSE passthrough

Secrets are generated by `deploy/gen-secrets.sh` into a gitignored `.env` file.

Helm chart is deferred to v1.1.

## Reverse proxy / TLS

KeenySpace does not terminate TLS. The server listens on plain HTTP. TLS termination belongs
to the reverse proxy layer (Caddy or nginx). The proxy must:

- Forward `X-Forwarded-Proto`, `X-Forwarded-Host`, `X-Forwarded-For`.
- Disable response buffering for SSE and StreamableHTTP (`flush_interval -1` in Caddy;
  `proxy_buffering off` in nginx).
- Set timeouts long enough for compile operations (10 minutes is safe; 600s in Caddy transport).

## Backup and restore

Source-of-truth backup is a pair:

- **FS root** — `tar` snapshot of the workspace directory tree.
- **Postgres** — `pg_dump`.

Atomicity is not required: pages can be regenerated by compile from WAL entries. At worst,
the most recent uncompiled batch is re-processed on next compile pass.

The `keenyspace backup` and `keenyspace restore` CLI commands automate this pair. The
`keenyspace doctor` command validates FS/Postgres registry consistency.

## Observability

- `/healthz` — liveness (process alive).
- `/readyz` — readiness (FS root writeable, Postgres reachable, IdP discovery endpoint reachable).
- Structured logs — JSON to stdout via structlog. Ingested by promtail or any log shipper.
- `/metrics` — Prometheus-compatible metrics endpoint. Scraped by Prometheus.
- Opt-in observability compose addon at `deploy/observability.yml`: Loki + Grafana + Prometheus + promtail.

## Secrets

KeenySpace accepts sensitive configuration (database URL, OIDC client secret, etc.) via
environment variables or env-file references. There is no built-in secret store; operators
integrate with their own solution (HashiCorp Vault, sops, dotenv) as needed.
